La anatomía de un proyecto Claude-ready

Así se ve un repo bien configurado: memoria, reglas, agentes y skills. Esta es la plantilla que construimos en vivo.

claude-code-starter/
claude-code-starter/ ├── CLAUDE.md # memoria del proyecto ├── README.md ├── .claude/ │ ├── settings.json # permisos + hooks │ ├── agents/ # subagentes │ │ ├── code-reviewer.md │ │ ├── test-writer.md │ │ └── explorer.md │ ├── skills/ # flujos reutilizables │ │ ├── new-component/SKILL.md │ │ ├── log-triage/SKILL.md │ │ └── pr-description/SKILL.md │ └── rules/ # reglas siempre activas │ ├── coding-style.md │ ├── testing.md │ └── security.md └── docs/ # "Second Brain" ├── architecture.md └── decisions.md
.claude/agents/code-reviewer.md
---
name: code-reviewer
description: Revisa diffs antes del
  commit (bugs, seguridad, reglas).
tools: Read, Grep, Glob, Bash
---
Eres un revisor senior. Reporta solo
hallazgos accionables por severidad...
.claude/skills/new-component/SKILL.md
---
name: new-component
description: Crea un componente React
  con las convenciones del repo.
---
1. Ubícalo en web/src/components/...
2. Archivos: .tsx, .test.tsx, index.ts

Laboratorio en vivo

El paso a paso que ejecutamos juntos

De carpeta vacía a una app con base de datos real conectada. Sigue cada paso en tu propia máquina — copia y pega los comandos.

1

Crear el proyecto

Una carpeta limpia con control de versiones desde el minuto cero.

PS> mkdir mi-app # o usa el proyecto de muestra PS> cd mi-app PS> git init
"Empezamos como cualquier proyecto serio: carpeta + git."
2

Lanzar Claude Code

Entra el copiloto. Lo primero NO es codear — es darle memoria.

PS> claude
3

Darle memoria: el CLAUDE.md

Genera el archivo que Claude leerá en cada sesión con el stack y las reglas.

/init
Lee el CLAUDE.md y las reglas en .claude/rules/. Resúmeme en 5 líneas qué es este proyecto, el stack y bajo qué reglas vas a trabajar.
"El CLAUDE.md separa a quien usa la IA de quien la domina."
4

Crear una Skill y un Agente

Empaqueta un flujo repetible y un especialista que revise tu código.

Crea la skill .claude/skills/new-component/SKILL.md que cree componentes con mis convenciones. Y un agente .claude/agents/code-reviewer.md que revise el diff antes del commit.
"Skills = CÓMO repetir un flujo. Agents = QUIÉN hace una tarea aislada."
5

Plan Mode antes de codear

Presiona Shift + Tab hasta ver "plan mode on". Pide el plan, no el código.

ultrathink: planifica la primera feature. No escribas código todavía, muéstrame el plan.
"Primero el plan, lo reviso, y recién ahí construye. Nunca al revés."
6

Construir usando la Skill

Aprueba el plan, activa Shift + Tab → "accept edits on" para no frenar.

Perfecto, impleméntalo siguiendo las reglas. Usa la skill new-component para la UI. Tests primero.
7

Conectar PostgreSQL (.env)

La contraseña va en .env (protegido por .gitignore), nunca en el repo.

# archivo .env DATABASE_URL="postgresql://postgres:123@127.0.0.1:5432/taskflow?schema=public"
PS> npx prisma migrate dev --name init
En Windows usa 127.0.0.1, no localhost (IPv6 a veces no engancha).
8

Ver la base de datos en vivo

Una UI en el navegador con tus tablas reales. Puedes agregar registros en vivo.

PS> npx prisma studio # abre http://localhost:5555
9

Correr la aplicación

PS> npm run dev # API :3000 · Web :5173
10

Revisar con el Agente y cerrar el PR

El agente revisa con contexto fresco; cierras con un PR profesional.

Usa el agente code-reviewer para revisar todos los cambios. Reporta por severidad.
Haz un commit pequeño y redacta el PR usando la skill pr-description.
"De cero a app con base de datos real, conectada y revisada."

Trucos y errores comunes

Lo que aprendimos a la fuerza

Los detalles que ahorran horas y evitan sustos en vivo.

localhost vs 127.0.0.1 (Windows)

Si la base "no responde" aunque el servicio corra, usa 127.0.0.1 en vez de localhost: en Windows, localhost resuelve a IPv6 y muchos clientes no enganchan.

Nunca secretos en el repo

La contraseña vive solo en .env, y .env va en .gitignore. Comparte un .env.example sin la clave real.

Activa "accept edits"

Para no aprobar cada cambio en vivo, Shift + Tab hasta accept edits on. Pero revisa siempre el diff antes del commit.

Higiene de contexto

Entre tareas distintas usa /clear. El contexto es memoria finita: fresco y condensado rinde mejor que largo y contaminado.

Plan antes de código

Para cualquier cambio que toque más de un archivo, pide el plan primero (Shift + Tab → plan). Corriges el rumbo antes de gastar tokens.

No abuses de los MCP

Cada MCP cuesta tokens y ruido. Conéctalos solo cuando aportan; a veces un CLI directo es más limpio que un servidor MCP.
