Paso 00 · el cimiento

El contrato — la memoria que hace que todo encaje

Antes de escribir una línea de código, le das a Claude las reglas del juego en un CLAUDE.md. Estos 5 nombres son el idioma común: si el servidor y cada cliente los respetan, todo encaja sin tocar nada. Es la lección entera del curso en una tarjeta.

Cliente → Servidor

SendMessage(text)
SetTyping(isTyping)

Servidor → Cliente

ReceiveMessage(user, text, sentAt)
UserTyping(user, isTyping)
PresenceChanged(online[])
CLAUDE.md — pégalo tal cual
# MiniTeams — Guía para Claude

## QUÉ construimos
Un chat en tiempo real estilo Microsoft Teams, en un solo proyecto:
- Backend: ASP.NET Core (.NET) + SignalR.
- Cliente web: HTML + CSS + JavaScript (con el cliente JS de SignalR), servido desde wwwroot.
- Un solo canal compartido "general" para empezar.

## Contrato del Hub (NO cambiar los nombres)
- Ruta: /hub/chat   ·   Identidad por query string: /hub/chat?user=<nombre>
- Cliente → Servidor:
  - SendMessage(string text)
  - SetTyping(bool isTyping)
- Servidor → Cliente:
  - ReceiveMessage(string user, string text, string sentAt)   // sentAt en ISO 8601
  - UserTyping(string user, bool isTyping)
  - PresenceChanged(string[] online)

## Reglas
- C# moderno: file-scoped namespaces, async/await, pasar CancellationToken.
- El servidor mantiene en memoria la lista de usuarios en línea (por nombre).
- Nada de autenticación por ahora: el nombre viene en ?user=.
- Mantener el código simple y legible; es una clase en vivo.

## Comandos
- Correr:   dotnet run
- Correr (HTTP, mejor para la clase):  dotnet run --launch-profile http

## Gotcha (Windows)
- Si algo "no conecta", usa 127.0.0.1 en vez de localhost (IPv6).
Sesión 02

El Mini-Teams — de cero a chat en vivo

Cinco prompts: el motor en tiempo real, la cara estilo Teams, presencia, historial y publicarlo para todos.

01
Backend · SignalR

El corazón en tiempo real

Un servidor que reparte cada mensaje al instante a todos los conectados. Aún sin interfaz, pero el hub ya vive.

pega en Claude Code · Plan Mode
Crea un proyecto ASP.NET Core (dotnet new web) llamado MiniTeams con SignalR.
Implementa un ChatHub en /hub/chat siguiendo EXACTAMENTE el contrato del CLAUDE.md:
- La identidad del usuario llega en ?user=<nombre> (léela en OnConnectedAsync).
- Mantén en memoria un conjunto de usuarios en línea (thread-safe).
- SendMessage(string text): reenvía a todos ReceiveMessage(user, text, sentAt-ISO).
- SetTyping(bool): reenvía a los demás UserTyping(user, isTyping).
- Al conectar/desconectar, emite PresenceChanged(online[]) a todos.
Registra SignalR en Program.cs y mapea el hub en /hub/chat. Sirve archivos estáticos desde wwwroot.
por quéSignalR mantiene un canal abierto (WebSocket) con cada navegador. Aquí el servidor habla primero: empuja el mensaje a todos sin que nadie tenga que preguntar. Esa es la diferencia con una API REST normal.
02
Cliente web

La cara estilo Teams

Una página que se ve como Teams y habla con el hub por SignalR sin recargar nunca. Abres dos ventanas y ya chatean.

pega en Claude Code
Crea el cliente web en wwwroot (index.html, styles.css, app.js) estilo Microsoft Teams:
- Una pantalla de "login" simple: pide un nombre y un botón Entrar.
- Layout: barra lateral con el canal "general" y la lista de usuarios en línea; a la derecha, el chat.
- app.js usa el cliente JS de SignalR (agrega <script> del CDN de @microsoft/signalr).
  - Conecta a /hub/chat?user=<nombre>.
  - Al enviar el formulario: llama SendMessage(text).
  - Escucha ReceiveMessage(user, text, sentAt) y pinta la burbuja (mis mensajes a la derecha).
  - Escucha PresenceChanged(online) y actualiza la lista de usuarios.
- Paleta estilo Teams (morados/índigo), burbujas redondeadas, tipografía limpia.
por quéEl cliente JS de SignalR usa el mismo protocolo que usaría un cliente .NET o móvil. Un solo hub, muchos clientes: tu navegador ya es un cliente en tiempo real.
03
Tiempo real · detalles

Presencia + “está escribiendo…”

Los dos detalles que hacen que se sienta vivo: el punto 🟢 de quién está en línea y el “…escribiendo” que todos conocemos.

pega en Claude Code
Agrega dos detalles en tiempo real:
1) Presencia: junto a cada usuario de la lista, un punto verde 🟢 si está en línea. Usa PresenceChanged.
2) "Escribiendo…": cuando escribo en la caja, llama SetTyping(true) (y SetTyping(false) al parar
   ~1.5s después de la última tecla). Escucha UserTyping(user, isTyping) y muestra
   "Fulano está escribiendo…" debajo del chat. Que desaparezca solo.
por quéEl “escribiendo…” es un evento efímero: no se guarda, es puro tiempo real. El truco del debounce de 1.5s (parar de avisar cuando dejas de teclear) es el mismo patrón de WhatsApp y Teams.
04
Persistencia · EF Core + SQLite

Que no se pierdan: historial

Ahora los mensajes solo viven en el aire. Los guardas para que al recargar sigan ahí — con SQLite, un archivo, cero instalación.

pega en Claude Code
Agrega persistencia con EF Core + SQLite:
- Paquetes: Microsoft.EntityFrameworkCore.Sqlite.
- Entidad Message (Id, User, Text, SentAt). DbContext con DbSet<Message>.
- Guarda cada mensaje en SendMessage antes de reenviarlo.
- Al conectar un cliente, envíale los últimos 50 mensajes (ReceiveMessage) como historial.
- Crea la base con EnsureCreated() al arrancar (sin migraciones, para la clase).
por quéSQLite es un solo archivo .db: sin servidor, sin cadenas de conexión raras. Perfecto para clase. En producción se puede cambiar a SQL Server sin tocar el resto.

Publicar · IIS

Publicarlo para todos

Para que se lo lleven: lo preparas para el servidor y todos entran por una URL, desde su casa.

pega en Claude Code

Sesión 03

El colega con IA — Nova

No tocamos el Mini-Teams. Le enchufamos una IA como un cliente más: misma conexión, mismo contrato — solo que su cerebro es Claude.

06
Nova · se conecta

Nova entra y escucha

Primero Nova entra al canal y oye todo — todavía sin cerebro. Aparece 🟢 en línea porque habla el mismo contrato que el simulador.

pega en Claude Code · Plan Mode
Crea un proyecto de consola .NET 10 llamado ColegaIA (carpeta colega-ia/) que se conecte
al hub del Mini-Teams como un cliente más, usando Microsoft.AspNetCore.SignalR.Client 10.0.0.
NO referencia la app: solo habla el contrato por su nombre (como el simulador).
- Identidad: se conecta a /hub/chat?user=Nova.
- Acepta el argumento --api <url> (por defecto http://127.0.0.1:5210).
- Escucha ReceiveMessage(user, text, sentAt) y, por ahora, imprime en consola "user: text".
- Ignora sus propios mensajes (cuando user == "Nova").
- Guarda en memoria las últimas 30 líneas del canal (una cola thread-safe) — las usaremos como contexto luego.
Al arrancar, imprime "Nova en línea" y quédate escuchando hasta Ctrl+C.
por quéEs 1:1 con el simulador de la S02: misma conexión, mismo ?user=. Nova ya sale en la lista de en línea del navegador porque habla el contrato — sin tocar la app.
07
El cerebro · API de Claude

Primera llamada a Claude

Cuando alguien nombre a Nova, en vez de una frase fija le preguntas a Claude — con la API cruda, sin SDK, para ver la forma real.

pega en Claude Code
Agrega el "cerebro" de Nova con la API de Mensajes de Claude, con HttpClient (SIN SDK):
- Endpoint: POST https://api.anthropic.com/v1/messages
- Headers: x-api-key (leído de la variable de entorno ANTHROPIC_API_KEY, o de --key),
  y anthropic-version: 2023-06-01.
- Modelo: claude-haiku-4-5-20251001 (o --model / variable CLAUDE_MODEL). max_tokens 400.
- Dispara SOLO cuando el mensaje mencione "Nova" (o "@nova"), y nunca a sus propios mensajes.
- System prompt: "Eres Nova, una compañera de equipo con IA en un chat estilo Teams de PeopleWorks.
  Respondes en español, cálida y BREVE (1-3 frases, como un chat)."
- En el mensaje de usuario, incluye las últimas líneas del canal (contexto) + lo que escribió la persona.
- Toma el texto de la respuesta de Claude y publícalo con SendMessage.
- Si la API falla, que Nova diga algo amable en el canal en vez de caerse.
por quéLa forma real de la IA: un system, unos messages, un JSON. Eso es todo. Y la clave (x-api-key) va en una variable de entorno, nunca en el código.
08
Naturalidad

Modales humanos

Nova responde de golpe y se nota que es máquina. Le das modales: que muestre “Nova está escribiendo…” mientras piensa, y que no conteste como loca.

pega en Claude Code
Dale a Nova modales humanos:
1) Antes de pensar, llama SetTyping(true); cuando publique la respuesta, SetTyping(false).
   Así el "Nova está escribiendo…" del Mini-Teams se enciende de verdad mientras Claude piensa.
2) Anti-spam: que no responda más de una vez cada ~2 segundos, y que NO procese dos respuestas
   a la vez (si ya está pensando, ignora el nuevo disparo).
por quéEl mismo indicador de “escribiendo” que tú programaste la semana pasada ahora lo dispara una IA. SetTyping es del contrato: Nova lo usa igual que cualquier cliente. Todo encaja.
09
Tool use · IA + tus datos

Manos para tus datos

Nova solo sabe lo que oyó desde que entró. Le das una herramienta para buscar en el historial real — la misma SQLite de la app.

pega en Claude Code
Dale a Nova una herramienta con "tool use" de Claude:
- Declara en el request un tool "buscar_historial" con input { termino: string, limite?: int }.
- Cuando la respuesta de Claude tenga stop_reason == "tool_use", ejecuta la herramienta y
  devuélvele el resultado como un bloque tool_result, y vuelve a llamar a la API (lazo) hasta
  que conteste con texto. Pon un tope de 4 vueltas por seguridad.
- La herramienta lee la MISMA base SQLite de la app en solo lectura (Microsoft.Data.Sqlite,
  Data Source=<db>;Mode=ReadOnly): SELECT User, Text FROM Messages WHERE Text LIKE '%termino%'
  ORDER BY Id DESC LIMIT <limite (máx 20)>. Devuelve las líneas encontradas.
- Acepta --db <ruta> (por defecto ..\app-referencia\miniteams.db). Si la base no existe,
  que la herramienta degrade con gracia (que Nova diga que no puede ver el historial ahora).
por quéEl lazo: pregunta → Claude pide una herramienta → tu código la ejecuta → Claude ve el resultado → responde. Eso que hicimos a mano es, por dentro, lo que te da empaquetado un servidor MCP.

Remate visual · reacciones

Reacciones que llueven — servidor

Una palabra nueva en el contrato: un toque 👏 y ese emoji lo ven TODOS. Primero el servidor lo reparte.

pega en Claude Code · servidor
Agrega al ChatHub un método SendReaction(string emoji) que reparta a TODOS (Clients.All)
un evento ReceiveReaction(user, emoji). NO lo guardes en la base: es efímero. Limita el emoji
a pocos caracteres por seguridad.
por quéUna función nueva es una palabra nueva en el contrato: la agregas una vez en el hub y todos los clientes pueden recibirla. Efímera, no toca la base.

Reacciones que llueven — cliente

Ahora la magia visible: emojis que flotan por la pantalla de todos, un aplausómetro y, al tope, ¡confeti!

pega en Claude Code · cliente
Añade una barra de botones de emoji (👏 ❤️ 🎉 🔥 😂). Al tocar uno, invoca SendReaction(emoji).
Escucha ReceiveReaction y haz "flotar" ese emoji subiendo por la pantalla (posición y deriva al

12
Remate visual · videollamada

Videollamada 1:1 — señalización

El mismo hub, ahora te conecta una llamada. El servidor solo presenta a los dos: dirige avisos a UNA persona.

pega en Claude Code · servidor
Agrega señalización de videollamada 1:1 al hub. Registra un IUserIdProvider que use el ?user=
como identidad (para poder dirigir eventos con Clients.User(nombre)). Métodos del cliente al
servidor: CallUser/AcceptCall/DeclineCall/HangUp(targetUser) y SendSignal(targetUser, signal).
Reenvían al destinatario: IncomingCall/CallAccepted/CallDeclined/CallEnded(fromUser) y
ReceiveSignal(fromUser, signal). El hub NO transporta vídeo: solo reenvía "sobres".
por quéAquí la señal va dirigida a una persona (Clients.User(nombre)), no a todos. El hub que reparte mensajes es el mismo que arma la llamada — solo hacía falta un mensajero.
13
Remate visual · videollamada

Videollamada 1:1 — WebRTC

El cierre que deja la boca abierta: vídeo en vivo dentro del Mini-Teams. El vídeo va directo entre navegadores (P2P).

pega en Claude Code · cliente (call.js)
Botón 📞 junto a cada usuario en línea. Al llamar, usa WebRTC (RTCPeerConnection con el STUN
de Google) para vídeo+audio P2P; intercambia la oferta/respuesta SDP y los candidatos ICE por
SendSignal/ReceiveSignal del hub. Muestra un panel con el vídeo remoto grande, tu vídeo en
pequeño (PiP) y botones de micro, cámara y colgar, más un aviso de llamada entrante con
Aceptar/Rechazar.
por quéWebRTC solo necesitaba un mensajero para presentar a los dos navegadores… y ya teníamos uno: el hub. El vídeo viaja directo entre ellos; el servidor no lo toca.
Nota: el vídeo exige contexto seguro — funciona en 127.0.0.1 y en https://.
